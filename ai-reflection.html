Input Materials Used During the Analysis

The analysis used several sources. The spec defined what the function should do. Research and notes added context, open questions, and edge cases that weren't in the spec.

  • 01.The original specification for WRITEBACKJOURNALDRAFT
  • 02.My own research into ERP systems, GL journals, writeback flows, and financial terminology
  • 03.Google Docs notes — questions, draft ideas, risks I spotted while reading the spec
  • 04.Miro brainstorming board — unclear behaviors, edge cases, and missing rules
  • 05.AI-assisted explanations of how similar systems may work under the hood
  • 06.Manual QA thinking on Excel behavior, ERP side effects, financial risks, and integration edge cases

The goal wasn't only to write test cases. It was to understand whether the requirement was complete, safe, and testable before writing a single step.

// SECTION 02
How I Used AI

Honestly — AI saved me a lot of the mechanical first-draft work. I used it to get up to speed on a domain I didn't know well, to turn scattered notes into something readable, and to get a first version of test cases fast enough that I could actually critique it rather than stare at a blank page.

What I asked AI to do
  • 01.Explain ERP, GL journals, and financial terminology in plain language
  • 02.Show how a writeback flow might work end-to-end under the hood
  • 03.Brainstorm test categories and surface potential risks before I wrote anything
  • 04.Find unclear or missing rules in the spec
  • 05.Draft the first version of test cases — so manual review had something concrete to start from
  • 06.Help merge near-duplicate cases and make the suite leaner
  • 07.Improve wording, restructure gaps, and organize the final artifact

I didn't take AI output at face value. Every result was reviewed and either kept as-is, rewritten, or thrown out. The thinking had to be mine — AI just gave me something to react to.

✗ what fell short

  • Kept the research as background context, not as final spec

Initial Test Brainstorming
> prompt used
I'm a QA engineer writing test cases for WRITEBACKJOURNALDRAFT.

Each call defines one journal line. Multiple calls form one batch.
The batch must always be On Hold. Must not post automatically.

Here is the full spec: [pasted]

Focus on: batch grouping, balance, required fields, ERP side effects,

✓ what worked
  • Quickly mapped the main test areas: happy path, validation, balance, ERP status, error handling
  • Spotted the naming mismatch between date/reportingPeriod and transactionDate/postPeriod
  • Suggested decimal precision risks — important for financial data
  • Treated WRITEBACKJOURNALDRAFT as a system integration, not just a formula
✗ what fell short
  • Accepted "at least one line required" literally — a single debit can't balance without a credit line
  • Didn't challenge the batch grouping key: branch? currency? postPeriod?
  • Assumed all Excel cells return the same batch number — spec doesn't confirm this
  • Some suggested cases were too generic, no concrete test data
→ what I did with it
  • Added gap analysis for grouping, return value scope, and selected range behavior
  • Rewrote vague assumptions as explicit gaps with test coverage links
  • Simplified the language so gaps are useful to QA, devs, and product

Formal Test Case Generation
> prompt used
Write formal test cases using this structure:
TC-ID · Category · Title · Preconditions · Test Data · Steps · Expected Result

No Priority or Severity. Simple English.
Numbered lists for Preconditions, Steps, Expected Result.
Preconditions specific to each case — not a shared header block.
Group duplicates. Smoke tests go at the end.
✓ what worked
  • Produced a clean, structured suite quickly
  • Covered the main flows: creation, On Hold status, no auto-posting, required fields, balance, duplicate prevention
  • Consistent structure across all 50 cases with separate columns
✗ what fell short
  • First version had too many near-duplicates — needed manual merging
  • Some preconditions were still vague ("valid ERP data exists")
  • Verification steps mixed into Steps instead of Expected Result
  • Called the field "Case Data / Variation" — renamed to Test Data
→ what I did with it
  • Merged near-duplicate cases into grouped scenarios
  • Rewrote preconditions to be specific per case
  • Moved all verification into Expected Result
  • Renamed fields, removed Priority/Severity entirely

// SECTION 08
Final QA Approach

Before writing test cases, the requirement itself had to be checked — whether it was complete, clear, and actually testable.

  • 01.Understand the business domain before writing any test cases
  • 02.Don't test only the happy path
  • 03.Check whether each requirement is actually testable as written
  • 04.Separate confirmed requirements from open assumptions
  • 05.Convert unclear behavior into explicit gaps
  • 06.Merge duplicates — a large suite is not the same as a good one
  • 07.Use concrete test data — not "valid inputs" but actual values
  • 08.Validate both Excel-side behavior and ERP-side effects
  • 09.Check that failed validation is visible to the user in Excel, not just logged somewhere
  • 10.Check that no invalid or partial data is written to ERP
  • 11.Treat duplicate batch prevention as a critical financial risk
  • 12.Treat auditability and traceability as product quality requirements, not extras

  • 03.Giving me a first structure to react to — instead of starting from nothing

10
// SECTION 10
Where AI Was Not Enough
AI was consistently weaker at
  • 01.Challenging the spec without being explicitly told to
  • 02.Detecting hidden product risks without direct guidance
  • 03.Avoiding assumptions when the spec left things open
  • 04.Deciding which test cases to merge vs. keep separate
  • 05.Understanding that Excel recalculation is dangerous when formulas have real ERP side effects

AI reads specifications charitably. It fills gaps with plausible assumptions rather than flagging them as risks. The most valuable part of this task required me to read carefully and ask: does this actually make sense?

Why I Decided to Create a Website

As the analysis grew, one flat document became too overloaded. There were test cases, gaps, assumptions, field specifications, validation rules, AI usage notes, and review comments.

Because of that, I decided that a small structured website would be a better format. It would let the reader switch between sections instead of scrolling through one long document.

Overall Assessment

AI was most valuable as a speed multiplier. It helped create the first draft faster — but the final QA value came from manual review. The parts that mattered required human judgment:

  • 01.Understanding what financial risks actually look like in an ERP context
  • 02.Challenging unclear or untestable requirements
  • 03.Deciding what is redundant and what must be tested separately
  • 04.Making sure errors are visible to the user in Excel, not just written to a log
  • 05.Separating confirmed behavior from open assumptions throughout
  • 06.Improving the artifact so it is useful for a real review, not just submitted
