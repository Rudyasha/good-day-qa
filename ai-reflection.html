AI Tool Reflection
Tool used: Claude (Anthropic) · Honest assessment of what worked and what didn't
AI was used as a thinking partner and first-draft generator — not as the final author. Every test case, assumption, and gap was reviewed, restructured, or rewritten by me. This section documents that process transparently.

Prompt 1 — Initial brainstorm

I'm a QA engineer writing test cases for a custom Excel function called
WRITEBACKJOURNALDRAFT. It creates draft GL journal batches in an ERP from
a spreadsheet. Each function call defines one line; multiple calls form
one batch. The batch always lands in "On Hold" status.

Here is the full spec:
[pasted entire spec text]

Give me a complete list of test categories I should cover, and flag any
ambiguities or gaps you see in the spec.

What AI did well:

  • Immediately identified the main test areas — happy path, required field validation, balance validation, edge cases — matching what an experienced QA engineer reaches for first.
  • Spotted the parameter name mismatch (date vs transactionDate) without me pointing it out.
  • Suggested testing floating-point precision for the balance check — a subtle but real risk that's easy to deprioritise.

What AI missed or handled poorly:

  • GAP-002 (empty line range) was not questioned. The AI accepted the acceptance criteria at face value and wrote a test case for it without asking how zero lines are even possible for a per-line function. I had to notice this myself.
  • GAP-003 (batch grouping) was skipped entirely. The AI never asked: "what if two calls have different batch descriptions — do they form one batch or two?" This is a core behavioural question.
  • GAP-008 (return value scope) was assumed, not questioned. The AI wrote "all cells return the batch ID" without flagging that the spec only says the function returns a value — it never says all cells return the same one.
  • The initial test case format was inconsistent — some had preconditions, some didn't; steps mixed numbered lists and prose.

Prompt 2 — Formalising test cases

Now write these as formal test cases using this structure:
TC-ID | Category | Priority | Title | Preconditions | Steps | Expected Result

Use High/Medium/Low priority. Mark edge cases where the spec is silent
with a note instead of a definitive expected result. Group by:
Positive, Required Fields, Balance Validation, Edge Cases.

What AI did well:

  • Produced a clean, structured table quickly — column layout, priority labelling, and grouping required minimal reformatting.
  • Correctly marked TC-024 through TC-027 (negative amounts, both-sides-populated) as "spec does not define this" rather than asserting pass/fail — the right call for undefined behaviour.
  • TC-022 (error message decimal precision) was a good catch — verifying that the error shows exactly 2 decimal places is easy to overlook.

What AI missed or handled poorly:

  • TC-002 used placeholder syntax that wouldn't work in a real Excel formula. AI wrote omitted params as ,, — valid in concept but the exact syntax varies by Excel version. I rewrote those steps explicitly.
  • Preconditions were vague — e.g. "ERP connected" without specifying account IDs or ledger names needed. I added those specifics.
  • TC-007 (currency defaulting to ledger currency) was missing until I asked separately — it's an explicit spec requirement that AI overlooked.
  • Priority was inconsistent: AI marked TC-028 (large numbers) as High; I downgraded it to Low — it's a boundary case, not a core flow.

What I would change about the prompts

  • Separate gap analysis from test case generation. Asking for both at once made AI rush the gaps to get to the test cases. Next time: a dedicated "find every ambiguity" prompt first, then a separate "write test cases" prompt.
  • Give AI a sceptical persona. Something like: "You are a QA engineer who has been burned by spec assumptions before. Flag anything the spec assumes but doesn't prove." This produces more critical output.
  • Ask it to challenge acceptance criteria, not just test against them. The "at least one line required" criterion was accepted uncritically. A prompt like "question whether each AC is testable as written" would surface issues like GAP-002 automatically.
  • Request explicit test data upfront. "Give me concrete input values for each test case" would have prevented the vague preconditions I fixed manually.

Overall assessment

AI was most valuable as a speed multiplier for structure — turning a mental list of ideas into a formatted table in seconds. It saved roughly 30–40 minutes of boilerplate work.

AI was least useful for critical thinking about spec coherence. It reads specifications charitably and fills gaps with plausible assumptions rather than surfacing them as risks. The most valuable part of this task — identifying GAP-002, GAP-003, and GAP-008 — required me to read the spec carefully and ask "does this actually make sense?"

The practical lesson: use AI to generate the first 80%, then apply domain knowledge and scepticism to the remaining 20% — which is often where the real testing value lives.
