01

Initial brainstorm

Goal: discover coverage areas and surface ambiguities before writing cases.

View exact prompt and full specification context 
I'm a QA engineer writing test cases for a custom Excel function called
WRITEBACKJOURNALDRAFT. It creates draft GL journal batches in an ERP from
a spreadsheet. Each function call defines one line; multiple calls form
one batch. The batch always lands in "On Hold" status.

Here is the full spec:

Task: Write test cases for the custom Excel function WRITEBACKJOURNALDRAFT

Glossary
A batch is a group of journal lines (transactions) that have the same
transaction date, ledger, and batch description.

A reporting period is a predefined accounting period where the transaction
date belongs (i.e. "Month Ending 31 March 2027").

Custom Function Description
As an ERP user, I want to create draft GL journal entries from Excel so that
I can prepare transactions for manual review and posting by an accountant.

WRITEBACKJOURNALDRAFT is an Excel custom function for creating draft GL journal
batches in the ERP directly from a spreadsheet range. Each function defines a
single line. Several of those functions placed together on a worksheet can
define one GL journal batch. The function never posts the batch — it always
lands in the On Hold status, and must be reviewed and posted manually inside
the ERP system.

Function Signature
=WRITEBACKJOURNALDRAFT(
  // batch properties
  connectionName,
  date,
  reportingPeriod,
  batchDescription,
  branch,
  Ledger,
  // line properties
  currency,
  account,
  debit,
  credit,
  lineDescription
)

Parameters
connectionName     string   Required: Yes  ERP connection name
transactionDate    date     Required: Yes  Date of the journal batch
postPeriod         string   Required: Yes  Financial period (e.g. "05-2026")
batchDescription   string   Required: No   Batch-level description
branch             string   Required: No   Branch identifier
ledger             string   Required: Yes  Target ledger
currency           string   Required: No   Currency code; defaults to ledger currency
account            string   Required: Yes  GL account per line
debit              number   Required: No   Debit amount per line
credit             number   Required: No   Credit amount per line
lineDescription    string   Required: No   Line description

Acceptance Criteria
Batch created on hold — The resulting batch in the ERP always has status
On Hold; no release or post action is triggered.

Balance validation — Total debits must equal total credits across all lines;
imbalanced input returns "Journal is not balanced: debits X.XX ≠ credits Y.YY"
error.

At least one line required — An empty line range returns
"At least one transaction line is required".

Return value — Returns "Draft batch #JE-00087 created" on success.

Give me a complete list of test categories I should cover, and flag any
ambiguities or gaps you see in the spec.

Handled well

  • Identified happy path, required fields, balance validation and edge cases immediately.
  • Spotted the parameter-name mismatch: date vs transactionDate.
  • Suggested floating-point precision checks for balance validation.

Missed or assumed

  • GAP-002: did not question how an empty range exists for a per-line function.
  • GAP-003: did not identify undefined batch-grouping behavior.
  • GAP-008: assumed all cells return the same batch ID without specification support.
  • Produced inconsistent formatting across early cases.
02

Formalising test cases

Goal: turn identified scenarios into a maintainable test-case format.

View exact prompt 
Now write these as formal test cases using this structure:
TC-ID | Category | Priority | Title | Preconditions | Steps | Expected Result

Use High/Medium/Low priority. Mark edge cases where the spec is silent
with a note instead of a definitive expected result. Group by:
Positive, Required Fields, Balance Validation, Edge Cases.

Handled well

  • Generated a structured test table quickly with usable grouping and labels.
  • Marked undefined negative/both-sides amount behavior as requiring clarification.
  • Suggested checking exactly two decimal places in the balance error.

Required my correction

  • TC-002: omitted-parameter syntax was conceptual rather than execution-ready Excel syntax.
  • Preconditions lacked concrete account, ledger and connection data.
  • Currency defaulting was missed despite being an explicit requirement.
  • Priority for large-number boundaries was overstated.
Prompt improvements
01

Separate analysis and generation

Run a dedicated ambiguity review before asking for test cases, so gaps are not rushed.

02

Demand scepticism

Ask the AI to challenge anything the specification assumes but does not prove.

03

Challenge acceptance criteria

Ask whether every criterion is testable as written, especially the empty-range requirement.

04

Require concrete data

Request runnable input values, setup and ERP assertions in the first case draft.

Overall assessment

Useful for velocity. Not a substitute for scepticism.

AI was most valuable as a speed multiplier for structure: it converted a mental coverage list into a formatted draft quickly and saved roughly 30–40 minutes of boilerplate work.

AI was least valuable for specification coherence: the key gaps concerning empty range, grouping and return-value scope required a human reviewer to ask, “does this actually make sense?”

Practical lesson: use AI for the first structured draft, then apply domain knowledge and scepticism to the high-risk decisions.
