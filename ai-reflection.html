How I Used AI

I used AI to speed up the parts that would otherwise consume the most time before any real QA thinking could begin.

AI was used to
  • Understand the domain and terminology from scratch
  • Break down how similar Excel-to-ERP writeback systems may work under the hood

  • Review whether test cases were too generic or too specific
  • Improve the field specification table, gaps, and assumptions

I did not use AI as a source of truth. I used it to generate ideas, challenge my own thinking, and accelerate the first draft. Each result was reviewed manually and then accepted, corrected, rewritten, or rejected.

Formal Test Case Generation
> prompt used
Write formal test cases using this structure:
TC-ID · Area · Title · Preconditions · Test Data · Steps · Expected Result

Do not use Priority or Severity.
Write simple English.
Use numbered lists for Preconditions, Steps, and Expected Result.
Make preconditions specific to each test case.
Do not repeat common setup unless the case actually needs it.
Group duplicated cases where possible.
Move smoke tests to the end.
✓ What worked
  • Produced a clean, structured test suite quickly
  • Covered the main flows: creation, On Hold status, no posting, required fields, invalid inputs, balance, duplicates
  • Helped convert a broad ideas list into 50 executable cases with separate columns
  • Maintained consistent structure across the entire suite
✗ What fell short
  • First version had too many near-duplicate cases that needed manual merging
  • Some preconditions were still too broad ("valid ERP data exists")
  • Some verification ended up in Steps instead of Expected Result
  • Field naming was inconsistent — "Case Data/Variation" vs "Test Data"
→ How I fixed it
  • Merged near-duplicate cases into grouped scenarios
  • Rewrote preconditions to be specific per case
  • Moved all validation into Expected Result
  • Renamed "Case Data / Variation" to Test Data throughout
  • Removed Priority and Severity — not needed for this artifact

Final QA Approach

The final approach was based on senior QA practice — not just test writing, but treating the requirement as something that must be challenged before it can be tested.

  • 01.Understand the business domain before writing test cases
  • 02.Do not test only the happy path
  • 03.Check whether each requirement is actually testable as written
  • 04.Separate confirmed requirements from open assumptions
  • 05.Convert unclear behavior into explicit gaps
  • 06.Connect major risks to test coverage
  • 07.Merge duplicates — a large suite is not the same as a good one
  • 08.Keep test cases executable: concrete test data, unambiguous expected result
  • 09.Validate both Excel-side behavior and ERP-side effects
  • 10.Check that failed validation is visible to the user in Excel
  • 11.Check that no invalid or partial data is written to ERP
  • 12.Treat duplicate batch prevention as a critical financial risk
  • 13.Treat auditability and traceability as part of product quality, not a nice-to-have

Where AI Was Not Enough
AI was weaker at
  • Challenging the specification without being directly told to do so
  • Detecting hidden product risks without explicit guidance
  • Avoiding assumptions when the spec left things open
  • Keeping preconditions specific instead of generic
  • Knowing which test cases should be merged vs. kept separate
  • Deciding what is truly critical for ERP financial writeback
  • Understanding that Excel recalculation is dangerous when formulas have real side effects

The most important QA value came from manual review, domain knowledge, and critical thinking. AI reads specifications charitably and fills gaps with plausible assumptions rather than surfacing them as risks.

At some point, the amount of material became too large for one linear document. The work covered requirement understanding, domain notes, gaps and assumptions, field specification, test cases, global validation rules, AI usage review, and notes from Google Docs and Miro.

Overall Assessment

AI was most valuable as a speed multiplier and structure assistant. It helped create the first draft faster, but the final QA value came from manual review. The most important parts required human judgment:

  • 01.Understanding the ERP domain and what financial risks actually look like in practice
  • 02.Challenging unclear or untestable requirements
  • 03.Deciding what should be tested per batch and what is redundant
  • 04.Checking whether errors are visible to the user in Excel — not just logged somewhere
  • 05.Making test cases specific and executable, not just structured
  • 06.Separating confirmed facts from open assumptions throughout
  • 07.Improving the final artifact so it is useful for a real review, not just a submission
The practical conclusion:
AI can generate a useful first draft, but a senior QA engineer must own the final logic, risk analysis, assumptions, and test coverage.

For this task, I fully invested in the process — I researched the domain, built separate notes in Google Docs and Miro, challenged weak assumptions, rewrote unclear parts multiple times, and changed the artifact structure when the material became too large for one document. The final result is not raw AI output. It is a reviewed QA artifact built through research, iteration, and critical analysis.
