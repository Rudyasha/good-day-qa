Spec Gaps & Assumptions
28 ambiguities identified — 25 senior-level + 3 unique originals · each with assumption and reasoning
Why this matters: Every untested assumption is a potential hidden defect. Making these explicit lets the team confirm with dev/BA before scripting — or add targeted test cases to probe each boundary.
G-01 · HIGH
Parameter naming conflict
The function signature uses date and reportingPeriod, but the parameter table names the same fields transactionDate and postPeriod. It is unclear which names are canonical — particularly relevant in error messages and documentation.
Assumption Assume the same fields by position mapping, but the spec must choose one source of truth before scripting. All test cases in this plan use the parameter table names (transactionDate, postPeriod).

Why: The parameter table is more detailed and structured — more likely to represent the final spec intent.
G-02 · CRITICAL
Trigger model missing — formula recalculation vs. explicit Writeback
The spec does not state whether the batch is created on formula recalculation (F9, workbook open, cell edit) or only on an explicit Writeback action. This directly determines whether recalculation creates duplicate drafts in ERP.
Assumption Only an explicit Writeback action creates a batch. Ordinary formula recalculation must not write to ERP.

Why: Automatic ERP writes on every recalculation would create uncontrolled duplicate journal drafts — a critical financial risk.
G-03 · CRITICAL
Idempotency not defined
Retry, timeout, workbook reopen, or pressing F9 may trigger the same operation multiple times. The spec does not describe any mechanism to prevent duplicate draft batches from being created.
Assumption The backend enforces an idempotency key per operation. Retrying the same operation with the same key returns the original batch, not a new one.

Why: Without idempotency, a network timeout followed by a retry would create two identical GL journal drafts — a serious financial data integrity issue.
G-04 · HIGH
Grouping key incomplete
The glossary defines a batch as lines sharing "same transaction date, ledger, and batch description," but does not specify whether connectionName, postPeriod, branch, or currency also contribute to the grouping key. Two calls with different branches could be grouped into one batch or separated.
Assumption Full grouping key: connectionName + transactionDate + postPeriod + batchDescription + branch + ledger + currency. All must match for lines to form one batch.

Why: The glossary definition is a minimum; standard ERP logic uses the full header to distinguish batches.
G-05 · CRITICAL
Balance scope unclear — per batch or global
The spec says "across all lines" but does not clarify whether balance is checked per grouped batch or globally across the entire selected range. If global, two imbalanced batches could incorrectly pass by cancelling each other out.
Assumption Balance is validated independently per grouped batch. The global total across all lines must not be used as a pass condition.

Why: Per-batch balancing is standard double-entry accounting. Global-only checking is financially incorrect and could hide real imbalances.
G-06 · HIGH
Batch vs. voucher level balance unclear
Some ERP systems validate balance at the voucher or journal entry level, not the batch level. The spec does not specify which granularity applies.
Assumption Balance is validated at the batch level unless the ERP defines a lower-level voucher construct.

Why: The spec describes batches as the primary container, so batch-level is the natural assumption.
G-07 · HIGH
Debit/credit mutual exclusion rule missing
Both debit and credit are marked optional. The spec does not state whether a single line can carry both a non-zero debit and a non-zero credit simultaneously, or whether a line with neither is valid.
Assumption Each line must have exactly one financial side: either debit > 0 or credit > 0, never both and never neither.

Why: Standard double-entry accounting places only one side per line. Both sides on one line produces a self-cancelling entry, which is meaningless and likely an input error.
G-08 · HIGH
Minimum line rule is too weak
The acceptance criteria states "at least one transaction line is required," but a single-line batch can never be balanced (debits would never equal credits). The rule does not require at least one debit line and one credit line.
Assumption A valid batch requires at least one debit line and at least one credit line. A batch with only one side will always fail the balance check.

Why: The "at least one line" criterion is logically incomplete — a single debit line with no credit line can never pass balance validation.
G-09 · MEDIUM
Date format not defined
Excel dates are locale-sensitive. A text value like "05/06/2026" is ambiguous — it means May 6 in US locale and June 5 in European locale. The spec does not define accepted date formats or how locale conversion is handled.
Assumption Accept Excel DATE values or DATE() formula results and normalize to YYYY-MM-DD. Text date strings are either rejected or handled only by a documented locale conversion rule.

Why: Silent locale conversion of text dates is a high risk for incorrect accounting dates in international deployments.
G-10 · HIGH
Period format inconsistent across spec sections
The glossary example uses "Month Ending 31 March 2027" while the parameter table example uses "05-2026". These are two different formats. The canonical format is not declared.
Assumption The parameter table format MM-YYYY (e.g. "05-2026") is canonical. The spec must document the accepted format and validate against ERP period codes.

Why: The parameter table example is more precise and machine-parseable. The glossary example reads as a human-readable description, not an input format.
G-11 · HIGH
Date/period consistency rule missing
The spec requires both transactionDate and postPeriod but does not state what happens when the date falls outside the given period — e.g. date=2026-03-15 but postPeriod="05-2026".
Assumption transactionDate must fall within the date range of postPeriod. A mismatch returns a validation error.

Why: Posting a March transaction to a May period is a common accountant mistake. The system must guard against it.
G-12 · HIGH
Closed period behavior not defined
The spec does not describe what happens when the specified postPeriod is closed or locked in the ERP. A draft could silently land in a closed period or be rejected — neither behavior is documented.
Assumption The ERP rejects creation into a closed or locked period with a clear error message.

Why: Writing to a closed period could corrupt finalized financial statements — a critical compliance risk.
G-13 · HIGH
Ledger allowed values not defined
ledger is typed as string but is ERP master data. The spec does not specify whether the value is a code, display name, or internal ID. Invalid and inactive ledger handling is not described.
Assumption Accept an active, accessible ERP ledger code or a specified identifier. Inactive, nonexistent, or inaccessible ledgers return a validation error.

Why: Ledger is a critical routing field — posting to the wrong ledger corrupts financial reporting.
G-14 · HIGH
Account validation rules missing
The spec does not describe how account is validated against the ERP chart of accounts. Inactive, restricted, or nonexistent account codes could be submitted without a defined outcome.
Assumption Account must exist, be active, and be accessible to the current user. Any deviation returns a validation error identifying the affected account.

Why: Journal entries to invalid accounts produce incorrect financial data that is difficult to audit and reverse.
G-15 · MEDIUM
Currency default resolution is incomplete
The spec states that currency defaults to the ledger's base currency, but does not specify where the default is resolved — in the Excel add-in, the connector, or the ERP — or how this is surfaced to the user.
Assumption The ERP or connector resolves the ledger default currency. The resolved currency is stored on the batch, not left empty.

Why: An empty currency field in ERP could cause incorrect multi-currency valuation or reporting.
G-16 · HIGH
Multi-currency behavior within one batch undefined
The spec does not state whether lines within a single batch can have different currencies, or whether a USD debit can be balanced against a EUR credit.
Assumption All lines within one batch must share the same currency unless the ERP explicitly supports currency conversion within a batch. Mixing currencies without conversion produces an invalid batch.

Why: Standard GL journals require monetary amounts to be in a consistent currency within a batch for the balance check to be meaningful.
G-17 · MEDIUM
Branch behavior undefined
branch is optional, but the spec does not specify whether it is a batch-level or line-level attribute, whether omitting it is valid for all ERP configurations, or whether it participates in the grouping key.
Assumption Branch is a batch-level attribute. If provided, it is validated against ERP master data and included in the grouping key. If omitted, the default branch or no-branch option is applied.

Why: Branch is typically a batch-level routing dimension in ERP. Its inclusion in the grouping key prevents mixing transactions across branches in one batch.
G-18 · CRITICAL
Atomicity behavior missing
The spec does not define what happens when one line within a batch fails validation or the ERP fails mid-operation. A partial batch (some lines written, some not) could corrupt the GL.
Assumption One Writeback operation is all-or-nothing. If any line or any batch fails, nothing is written to ERP. No partial batches are created.

Why: A partial GL journal creates an imbalanced financial record that may not be detectable until close-of-period — a serious audit risk.
G-19 · CRITICAL
Timeout and retry behavior not defined
If the ERP creates a batch but the response times out before Excel receives it, the user may retry — creating a duplicate batch. The spec does not describe timeout handling or safe retry semantics.
Assumption Retry with the same idempotency key returns the original batch result without creating a new one. Excel must not show a false success or false failure during timeout.

Why: Financial duplicate batches from retry storms are a critical data integrity issue in ERP systems.
G-20 · HIGH
Permission model not described
The spec does not describe the permission model. It is unclear what ERP roles are required to create journal drafts, whether per-ledger or per-branch permissions exist, or how permission errors are surfaced.
Assumption ERP permissions are enforced server-side. A user without sufficient access receives a clear permission error. No partial success for partially-accessible data.

Why: Unauthorized financial writes are a compliance and audit risk. Permissions must be enforced at the ERP level, not only client-side.
G-21 · HIGH
Audit trail and telemetry not specified
The spec does not describe whether successful or failed operations are logged, what is included in the audit trail, or how support engineers would trace a failed Writeback operation.
Assumption Each operation is assigned a correlation ID. Successful creations and validation failures are logged. The audit trail records user, timestamp, source (Excel/add-in), and outcome.

Why: Without audit logs and correlation IDs, investigating a GL error or duplicate batch in production is extremely difficult.
G-22 · HIGH
API/OData contract not described
The spec does not define the REST/OData payload format, response schema, or error code mapping between the ERP API and Excel. This makes it impossible to write API-level tests or verify correct field mapping.
Assumption A normalized REST/OData or API contract exists (likely JSON). Field names, data types, required fields, and error response format must be documented before integration testing can begin.

Why: Without a contract, API validation tests would be guessing at field names and error codes.
G-23 · HIGH
Cross-platform behavior not defined
The spec does not address differences between Excel for Windows, Excel for macOS, and Excel Web. Date parsing, decimal separators, and add-in capabilities differ across platforms.
Assumption Behavior should be consistent across Windows, macOS, and Web, or platform-specific limitations should be explicitly documented.

Why: A date that parses correctly on US Windows may shift by one month on a European Mac due to locale settings — a silent accounting error.
G-24 · MEDIUM
Queue/background job processing is ambiguous
It is unclear whether the Writeback operation is synchronous (ERP responds before Excel unblocks) or asynchronous (a background job or queue processes the request). Async processing creates additional risks around duplicate messages and poison queue entries.
Assumption Assume near-real-time REST/OData synchronous processing. If async is used, timeout/retry/job-status behavior must be defined along with idempotency and poison message handling.

Why: Async without explicit retry semantics and idempotency leads to duplicate batch creation on transient failures.
G-25 · MEDIUM
Field length limits and numeric precision not specified
No maximum lengths are defined for string fields (batchDescription, lineDescription, account, etc.) and no numeric precision rules are defined for debit and credit (e.g. max 2 decimal places, max value).
Assumption String fields: likely 255 chars max. Numeric fields: likely decimal(19,4) or decimal(19,2). Maximum line count per batch is limited. All limits must be defined and enforced with clear error messages.

Why: Undefined limits lead to data truncation, overflow errors, or silent corruption — all of which are difficult to diagnose in production.
GAP-005 · ORIGINAL
Zero-value lines — are they valid?
The spec does not define what happens when both debit and credit are 0 (or both omitted entirely). A zero-amount line is technically valid in some ERPs but meaningless in practice. Neither G-07 nor G-08 address this specific case.
Assumption A line with debit=0 and credit=0 is rejected with a validation error — zero-value lines carry no financial meaning and should not be silently accepted.

Why: Accepting zero lines could produce empty journal entries that pass balance validation but clutter the GL. Strict rejection with a clear message is safer.
GAP-006 · ORIGINAL
Negative debit/credit values — no explicit definition
debit and credit are typed as number, but the spec does not state whether negative values are valid. Negative amounts are sometimes used for reversal entries in accounting. G-07 addresses "both populated" but not the sign of individual values.
Assumption Negative values are rejected with a validation error (e.g. "Debit amount must be a positive number"). If reversals are a supported use case, this must be explicitly documented with its own behavior.

Why: Without explicit support, accepting negatives could silently produce incorrect or reversed entries. Failing loudly is the safer default.
GAP-008 · ORIGINAL
Return value scope: does every line cell show the batch ID?
The spec says the function "returns 'Draft batch #JE-00087 created'". Since each WRITEBACKJOURNALDRAFT call occupies a separate Excel cell, it is unclear whether every cell returns this value, or only one (e.g. the first or last line evaluated). None of the PDF gaps address the per-cell return behavior.
Assumption All cells belonging to the same batch return the same success message with the same batch ID. Uniform return values avoid a confusing UX where only some cells show a confirmation.

Why: If only one cell returned the batch ID, the user would have no visual confirmation for the other lines — a poor and error-prone experience for accountants.
