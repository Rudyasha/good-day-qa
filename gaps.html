Spec Gaps & Assumptions
8 ambiguities identified — each with stated assumption and reasoning
Why this matters: Every untested assumption is a potential hidden defect. Making these explicit lets the team confirm with dev/BA before scripting — or add targeted test cases to probe each boundary.
GAP-001
Parameter name mismatch between signature and parameter table
The function signature uses date and reportingPeriod, but the parameter table names the same arguments transactionDate and postPeriod. It is unclear which names are canonical — particularly relevant in error messages and documentation.
Assumption The parameter table names (transactionDate, postPeriod) are authoritative. The signature likely reflects an earlier draft. All test cases in this plan use the table names.

Why: The parameter table is more detailed and structured — more likely to represent the final spec.
GAP-002
"Empty line range" is undefined for a per-line function
The acceptance criteria state: "An empty line range returns 'At least one transaction line is required'." But each WRITEBACKJOURNALDRAFT call defines exactly one line — the function has no range parameter. It is unclear how zero lines would be represented, or what triggers this error.
Assumption This error is triggered by a batch-level orchestration mechanism — e.g. when a user invokes a "create batch" action with no function calls present, or when all cells in the designated batch range are empty. TC-023 is marked as pending clarification on the trigger mechanism.

Why: The spec describes batch-level behaviour alongside per-line calls, so a separate batch-creation trigger likely exists.
GAP-003
Batch grouping logic is not specified
The glossary says a batch is lines sharing the same date, ledger, and batch description. But what happens when two calls on the same sheet have different batch descriptions or dates? Do they form separate batches in one execution, or is only one batch allowed per execution?
Assumption All calls sharing identical transactionDate, ledger, and batchDescription are grouped into one batch. Calls with different combinations create separate batches. Balance validation applies independently per group.

Why: The glossary definition directly implies grouping by these three fields, consistent with standard ERP batch logic.
GAP-004
Behaviour when both debit and credit are non-zero on the same line
Both debit and credit are optional. The spec does not state whether a single line can carry both a non-zero debit and a non-zero credit simultaneously. Standard double-entry accounting places only one side on each line.
Assumption Providing both non-zero debit and credit on the same line is rejected with a validation error. If accepted, only one value is used. This needs explicit verification (TC-024).

Why: Standard GL conventions prohibit dual-sided lines. An ERP would likely enforce this, but the spec must state it clearly.
GAP-005
Zero-value lines — are they valid?
The spec does not define what happens when both debit and credit are 0 (or omitted entirely). A zero-amount line is technically valid in some ERPs but meaningless in practice.
Assumption A line with debit=0 and credit=0 is accepted (contributes 0 to the balance sum) but may produce a warning in the ERP UI noting no financial impact.

Why: Strict rejection would frustrate users with placeholder lines. A warning is the safer, more user-friendly approach.
GAP-006
Negative debit/credit values — no definition
debit and credit are typed as number, but the spec does not state whether negative values are valid. Negative amounts are sometimes used for reversal entries in accounting.
Assumption Negative values are rejected with a validation error (e.g. "Debit amount must be a positive number"). If reversals are a supported use case, this must be explicitly documented.

Why: Without explicit support, accepting negatives could silently produce incorrect entries. Failing loudly is safer.
GAP-007
No validation rule for transactionDate vs. postPeriod mismatch
The spec requires both a transactionDate and a postPeriod. It does not state what happens when the date falls outside the given period — e.g. date=2026-03-15 but postPeriod="05-2026".
Assumption If transactionDate does not fall within the postPeriod range, the function returns a validation error. This is standard ERP behaviour preventing misclassified entries.

Why: Posting a March transaction into a May period is a common accountant mistake — the ERP and function should both guard against it.
GAP-008
Return value scope: does every line cell show the batch ID?
The spec says the function "returns 'Draft batch #JE-00087 created'". Since each call is a separate Excel cell, it is unclear whether every cell returns this value, or only one (e.g. the first or last line evaluated).
Assumption All cells belonging to the same batch return the same success message with the same batch ID. Uniformity avoids confusing UX where only some cells show a result.

Why: If only one cell returned the ID, the user would have no visual confirmation for the other lines — a poor and error-prone experience.
