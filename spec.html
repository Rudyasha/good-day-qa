Spec Review
Feature analysis · WRITEBACKJOURNALDRAFT · what the spec says and what it should say
◈ Feature Understanding

WRITEBACKJOURNALDRAFT is a custom Excel function that creates draft GL (General Ledger) journal batches in an ERP system directly from a spreadsheet. Each function call defines a single journal line. Multiple calls on the same worksheet can form one batch, grouped by shared header fields.

Core insight: This is not a simple Excel formula — it is a financial writeback integration between Excel and a Cloud ERP. The critical risks are not input validation edge cases: they are duplicate batch creation, wrong balance scope, incorrect ledger/period mapping, auto-posting, and timeout/retry behavior.

The function never posts. The created batch always lands in On Hold status and requires manual review by an accountant inside the ERP before posting. This constraint must be verified — not assumed.

◎ Key Terms
TermMeaning
BatchContainer document grouping journal lines that share the same transaction date, ledger, and batch description. In an ideal spec, the full grouping key must be defined precisely.
LineOne row inside a batch. Contains an account and exactly one financial side — either a debit or a credit amount, not both.
LedgerERP master data entity — a chart-of-accounts register. Not free text. The accepted identifier format (code, name, or internal ID) must be documented.
Debit / CreditThe two sides of a double-entry journal. Total debits across all lines in a batch must equal total credits before the batch can be created.
On HoldDraft ERP batch status. Saved but not posted or released. Requires manual accountant review. The function must never trigger posting or release.
postPeriodThe accounting period the transaction is attributed to (e.g. "05-2026"). The transaction date must fall within this period. Period must be open in ERP.
Idempotency keyA technical key that prevents duplicate batch creation when the same Writeback operation is retried after a timeout or network failure.
Correlation IDA unique trace identifier for an operation — used to link Excel events, API/OData logs, telemetry, and ERP audit trail entries for support and debugging.

◈ Risk-Based Priority Map
P0
Duplicate batch creation
Excel recalculation, workbook reopen, or timeout retry creates multiple identical GL journal drafts — a critical financial data integrity risk.
P0
Wrong status or auto-posting
Batch bypasses "On Hold" and gets posted/released automatically — journal entries appear in financial reports without accountant review.
P0
Wrong balance validation scope
Global-range balance check masks imbalanced individual batches — incorrect financial data passes silently.
P0
Partial batch creation (atomicity)
One invalid line leaves a partial batch in ERP — an imbalanced orphaned journal entry that is difficult to detect and reverse.
P0
Wrong ledger / account / period
Financial data posted to incorrect ledger or period corrupts reporting and may require audit-trail investigation to reverse.
P1
Date/locale conversion errors
Ambiguous text dates (05/06 = May 6 vs June 5) silently produce wrong accounting dates across platforms.
P1
Permission bypass
Unauthorized financial writes if ERP-side permissions are not enforced server-side.
P1
Missing traceability
No correlation ID or audit log entry means support cannot investigate ERP/API failures after they occur.
P1
Cross-platform inconsistency
Different date or decimal behavior between Windows / macOS / Excel Web produces incorrect journal data depending on the user's environment.
P1
Script / markup injection via text fields
Unsanitized batchDescription or lineDescription could execute in ERP UI — XSS or formula injection risk.

Out of Scope
  • Manual accountant posting workflow after draft creation
  • Internal ERP database schema (unless access granted)
  • Full infrastructure load and performance testing
  • Exact queue implementation (unless architecture confirms async)
  • ERP financial reporting after posting
  • Velixo add-in installation and licensing
  • ERP-side UI testing beyond batch/line inspection
